# Quantum Maze — Macro Roadmap Narrative v1

## 1. Scopo del documento

Questo documento definisce la macro-roadmap narrativa e strategica di Quantum Maze.

Non è una roadmap tecnica.
Non è una lista di feature.
Non è un piano di sprint.

È una bussola creativa.

Serve a mantenere coerenza tra:

* rituale;
* cicli;
* Observer;
* Threshold;
* Null Chamber;
* metamorfosi di Q;
* progressione del player;
* mistero centrale.

La direzione confermata è:

```text
Quantum Maze non è un arcade classico.
Quantum Maze non è un maze game tradizionale.
Quantum Maze non è un puzzle esplicito.

Quantum Maze è un rituale interattivo di raccolta, memoria, errore, soglia e metamorfosi.
```

Principio guida:

```text
Criptico sì, casuale no.
```

Il gioco può non spiegare tutto, ma deve avere una logica interna coerente.

---

## 2. Stato attuale: Spark Build v0.1

Baseline consolidata:

```text
Quantum Maze — Spark Build v0.1
```

Stato attuale:

* canvas runtime funzionante;
* menu iniziale;
* tutorial rapido;
* pausa avanzata;
* Q controllabile;
* Q accettata come biglia pulsante / entità contenuta;
* frammenti visibili e raccoglibili;
* progressione a cicli funzionante;
* ciclo 3/3 con tremore/drift come segnale di instabilità;
* Observer presente come entità/sistema;
* baseline salvata e protetta.

Interpretazione narrativa dello stato attuale:

```text
Q non è ancora pienamente rivelata.
Q è contenuta.
Q può evolvere.
```

Questa scelta è coerente con la futura Null Chamber: Q non deve essere visivamente definitiva all’inizio, ma trasformarsi progressivamente.

---

## 3. Struttura macro in quattro atti

La progressione prospettica di Quantum Maze può essere organizzata in quattro atti.

```text
ACT I   — Spark / Ritual Discovery
ACT II  — Threshold / Observer Inversion
ACT III — Null Chamber / Q Metamorphosis
ACT IV  — Transfer / Final Unknown
```

Questa struttura non deve diventare rigida o troppo esplicita nel gioco.
Serve internamente per guidare design, implementazione e priorità.

---

# ACT I — Spark / Ritual Discovery

## 4. Funzione dell’Act I

Act I è la fase in cui il player entra nel sistema.

Il player deve imparare:

* a muovere Q;
* a raccogliere frammenti;
* a osservare i cicli;
* a percepire l’Observer;
* a capire che il ciclo 3 è diverso;
* a sospettare che l’ordine dei frammenti conti.

Act I non deve spiegare la regola nascosta.

Deve far nascere il dubbio.

Domanda mentale desiderata:

```text
Perché il gioco reagisce così?
```

---

## 5. Cicli dell’Act I

La struttura dei cicli resta:

```text
Cycle 1/3 = Stabilizzazione
Cycle 2/3 = Memoria / Disturbo
Cycle 3/3 = Collasso / Rivelazione
```

### Cycle 1/3 — Stabilizzazione

Funzione:

* familiarità;
* controllo;
* raccolta;
* mondo relativamente stabile;
* Observer in osservazione.

Il player pensa:

```text
Sto imparando il sistema.
```

### Cycle 2/3 — Memoria / Disturbo

Funzione:

* primi segnali di memoria;
* primi sospetti di pattern;
* Observer più reattivo;
* messaggi meno neutri.

Il player pensa:

```text
Il gioco si sta ricordando qualcosa.
```

### Cycle 3/3 — Collasso / Rivelazione

Funzione:

* tremore;
* instabilità;
* pressione;
* rivelazione implicita della sequenza;
* sensazione che completare non equivalga a vincere.

Il player pensa:

```text
Qui c’è qualcosa da capire.
```

---

## 6. Regola nascosta dell’Act I

La sequenza canonica candidata è:

```text
UNSTABLE → HIDDEN → NORMAL → ECHO → UNSTABLE
```

Formula rituale:

```text
chiamare → vedere → stabilizzare → ricordare → aprire
```

Significato dei frammenti:

```text
UNSTABLE = contatto / disturbo / chiamata dell’Observer
HIDDEN   = rivelazione / verità nascosta
NORMAL   = stabilizzazione
ECHO     = memoria / ripetizione
UNSTABLE = chiusura del circuito / apertura della soglia
```

Il player non deve ricevere questa sequenza come tutorial.
Deve intuirla attraverso reazioni, messaggi e ripetizione.

---

## 7. Loop come vero inizio

Al primo completamento dei tre cicli, il player non deve ricevere una vittoria classica.

Direzione preferita:

```text
Fine Cycle 3/3 → TRANSFER DENIED → ritorno a Cycle 1/3
```

Possibili messaggi:

```text
TRANSFER DENIED
TRACE REPEATED
PATTERN REJECTED
CYCLE 1 INITIALIZED
```

Effetto desiderato:

```text
Ho completato i tre cicli, ma qualcosa è andato storto.
Cosa ho sbagliato?
```

Il loop deve trasformare il gioco da raccolta a decodifica.

---

# ACT II — Threshold / Observer Inversion

## 8. Funzione dell’Act II

Act II inizia quando il player non si limita più a completare i cicli, ma prova a ripetere correttamente la sequenza rituale.

Il player deve aver capito o intuito che:

* i frammenti hanno identità;
* l’ordine conta;
* l’Observer reagisce;
* il ciclo 3 mostra qualcosa;
* il loop non è una sconfitta casuale.

Domanda mentale desiderata:

```text
Posso essere riconosciuto dal sistema?
```

---

## 9. Successo rituale

Il successo non è una vittoria immediata.

Prima soglia di successo:

```text
Il player ripete correttamente la sequenza in Cycle 1/3.
Poi la ripete correttamente in Cycle 2/3.
All’inizio del Cycle 3/3 compare il Threshold.
```

Segnali possibili:

```text
THRESHOLD DETECTED
PATTERN RECOGNIZED
TRANSFER POSSIBLE
```

Observer:

```text
>>>
```

---

## 10. Significato di `>>>`

`>>>` è il passe-partout del sistema.

Non significa solo pericolo.

Significa:

```text
pattern riconosciuto
soglia possibile
accesso aperto
rituale allineato
presenza piena
```

`>>>` deve apparire solo quando il sistema riconosce un pattern significativo.

Il player deve imparare che:

```text
Quando compare >>>, qualcosa si è allineato.
```

---

## 11. Attraversamento del Threshold

Quando il player porta Q nel Threshold, non deve iniziare semplicemente un nuovo livello.

Deve avvenire un cambio di grammatica.

Sequenza suggerita:

```text
1. Q entra nel Threshold.
2. Il tremore si sospende.
3. Il mondo rallenta.
4. Observer mostra >>>.
5. >>> si inverte in <<<.
6. Il player percepisce che il viaggio sta iniziando.
```

Possibili messaggi:

```text
VECTOR REVERSED
RETURN SIGNAL
THE PATH BEGINS
TRACE ADMITTED
```

---

## 12. Observer Inversion: `<<<`

`<<<` è la forma definitiva o post-Threshold dell’Observer.

Prima del Threshold:

```text
Observer = osservatore / validatore / interferenza
```

Dopo il Threshold:

```text
Observer = direzione / richiamo / vettore di viaggio
```

Interpretazione:

```text
>>> = apertura
<<< = richiamo
```

Oppure:

```text
>>> = access granted
<<< = return vector detected
```

Il player non deve capire tutto.
Deve percepire che l’asse del gioco è cambiato.

---

# ACT III — Null Chamber / Q Metamorphosis

## 13. Funzione dell’Act III

Act III introduce la Null Chamber.

La Null Chamber non è:

* uno shop;
* una stanza premio;
* un menu upgrade;
* un nuovo livello arcade;
* una spiegazione.

È:

```text
uno spazio rituale di decompressione, ascolto, metamorfosi e identità.
```

Domanda mentale desiderata:

```text
In cosa può evolvere Q?
```

---

## 14. Atmosfera della Null Chamber

La Null Chamber deve essere l’opposto del Cycle 3/3.

Cycle 3/3:

```text
tremore
pressione
collasso
urgenza
instabilità
```

Null Chamber:

```text
lentezza
ovattamento
quiete
sospensione
sicurezza apparente
metamorfosi
```

Il player deve sentire:

```text
Sono al sicuro, ma non sono libero.
Il sistema mi concede tempo.
Questo momento serve a qualcosa.
```

---

## 15. Movimento nella Null Chamber

Dentro la Null Chamber, Q si muove più lentamente.

Il cambio di movimento deve comunicare:

```text
Qui le regole sono diverse.
```

Il movimento deve essere:

```text
più lento
più morbido
più ovattato
più denso
meno urgente
```

Questa lentezza deve invitare a osservare, ascoltare e provare.

---

## 16. I tre nuclei

La Null Chamber introduce tre nuclei sconosciuti.

Non devono essere spiegati come upgrade.

Devono apparire come tre possibilità misteriose di metamorfosi.

Il player deve poter:

* avvicinarsi a un nucleo;
* vedere Q cambiare temporaneamente;
* ascoltare una sonorità specifica;
* allontanarsi;
* provare un altro nucleo;
* confrontare sensazioni senza sapere i benefici esatti.

La logica è:

```text
Proximity preview → allontanamento → prova altro nucleo → permanenza → metamorfosi
```

---

## 17. Preview metamorfica

Ogni nucleo ha una soglia di prossimità.

Quando Q entra nel raggio del nucleo:

* il nucleo si attiva;
* Q mostra una variazione visiva temporanea;
* parte una firma sonora;
* l’effetto aumenta con la vicinanza o la permanenza.

Quando Q si allontana:

* il suono decade;
* il cambiamento visivo si attenua;
* il player può provare un altro nucleo.

La scelta non è immediata.
La scelta nasce dalla permanenza.

---

## 18. Commit metamorfosi

Solo dopo una permanenza più lunga vicino a uno dei nuclei, la metamorfosi si imprime.

Non deve essere un click.
Non deve essere un menu.

Deve sembrare un gesto rituale:

```text
Q resta.
Il nucleo risponde.
La camera si accorda.
La metamorfosi si imprime.
```

Il player non sceglie un potere.
Sceglie una possibilità di identità.

---


# ACT IV — Transfer / Final Unknown

## 22. Funzione dell’Act IV

Act IV non è ancora definito in modo conclusivo.

Deve restare aperto.

La direzione generale è che, dopo varie soglie e metamorfosi, il player scopre che il sistema non riguarda solo uscire dal maze.

Possibili domande finali:

```text
Q deve essere liberata?
Q deve essere trasferita?
Q è già stata trasferita?
L’Observer sta guidando Q o trattenendola?
Il player sta salvando Q o completando il protocollo del sistema?
```

---

## 23. Cosa non definire ancora

Non fissare troppo presto:

* finale;
* boss;
* spiegazione completa dell’Observer;
* origine esatta di Q;
* significato definitivo del sistema;
* numero finale delle Null Chamber;
* natura ultima del Transfer.

Il mistero deve restare vivo.

Act IV deve essere preparato, non chiuso.

---

## 24. Possibili assi futuri

Act IV potrebbe lavorare su:

```text
scelta finale
rifiuto del trasferimento
fusione con Observer
liberazione incompleta
loop consapevole
ritorno volontario
Q che diventa sistema
Observer che diventa Q
```

Questi sono solo assi esplorativi.

Non vanno ancora implementati né fissati.

# 25. Principi guida globali

## Non trasformare Quantum Maze in arcade generico

Evitare:

* nemici classici;
* power-up espliciti;
* livelli standard;
* punteggio come obiettivo primario;
* upgrade menu;
* spiegazioni lunghe;
* missioni guidate troppo chiare.

## Preservare la grammatica rituale

Ogni meccanica deve servire almeno uno di questi elementi:

```text
memoria
sequenza
errore
soglia
metamorfosi
richiamo
identità
```

## Mantenere il mistero decodificabile

Il player non deve essere confuso in modo casuale.

Deve poter pensare:

```text
Non capisco ancora, ma sento che c’è una logica.
```

## Procedere per micro-step

Ogni sviluppo futuro deve essere:

* piccolo;
* testabile;
* reversibile;
* coerente con Spark Build v0.1;
* protetto da commit/tag quando stabile.

# 26. Roadmap implementativa prospettica

Questa non è ancora una roadmap tecnica dettagliata, ma una sequenza logica di sviluppo.

## Phase 1 — Stabilizzare Spark

Obiettivo:

```text
mantenere stabile il core loop attuale
```

Attività:

* non rompere baseline;
* correggere bug minori;
* mantenere cicli funzionanti;
* consolidare Q come entità contenuta;
* evitare feature invasive.

## Phase 2 — Ritual Tracking

Obiettivo:

```text
registrare l’ordine dei frammenti raccolti
```

Attività future:

* tracciare sequenza per ciclo;
* confrontare con sequenza canonica;
* produrre feedback minimo;
* non cambiare ancora troppo la grafica.

## Phase 3 — Observer Feedback

Obiettivo:

```text
far reagire l’Observer alla sequenza rituale
```

Attività future:

* `>. .` per primo contatto;
* `>-. .` per desync/rivelazione;
* `>>.` per interferenza attiva;
* `>>>` per pattern riconosciuto;
* messaggi brevi e criptici.

## Phase 4 — Loop / Transfer Denied

Obiettivo:

```text
trasformare il completamento fallito in loop rituale
```

Attività future:

* fine Cycle 3 senza pattern corretto;
* messaggi TRANSFER DENIED / TRACE REPEATED;
* ritorno a Cycle 1;
* memoria implicita del tentativo.

## Phase 5 — Threshold

Obiettivo:

```text
aprire la prima soglia
```

Attività future:

* rilevare sequenza corretta in Cycle 1 e Cycle 2;
* mostrare Threshold a inizio Cycle 3;
* mostrare `>>>`;
* permettere a Q di attraversare.

## Phase 6 — Observer Inversion

Obiettivo:

```text
trasformare >>> in <<<
```

Attività future:

* sospendere tremore;
* invertire simbolo Observer;
* comunicare cambio di rotta;
* preparare ingresso nella Null Chamber.

## Phase 7 — Null Chamber I

Obiettivo:

```text
introdurre spazio lento di metamorfosi
```

Attività future:

* creare area ovattata;
* rallentare movimento Q;
* mostrare tre nuclei;
* aggiungere preview visive;
* aggiungere tre firme sonore;
* permettere test dei nuclei;
* imprimere prima metamorfosi.

## Phase 8 — Q Signature

Obiettivo:

```text
dare identità alla Q trasformata
```

Attività future:

* generare firma/rango;
* modificare leggermente aspetto Q;
* rendere la scelta significativa ma non spiegata;
* preparare ulteriori camere future.

---

# 27. Frase guida

La frase guida attuale del progetto è:

```text
Non stai cercando l’uscita dal labirinto.
Stai insegnando a Q a diventare qualcosa capace di attraversarlo.
```

Questa frase deve guidare le prossime decisioni narrative, estetiche e meccaniche.
